## Recommended screenshot story for bracketball

1. Home
   Show the mobile shell with clear app navigation and live product context.
2. Pools
   Show joined pools plus the discover/join flow.
3. Leaderboard
   Show the strongest seeded pool with visible standings and the `Share Invite` action.
4. Bracket
   Show the bracket view for the same pool.
5. Profile
   Show avatar, profile details, and notification controls.
6. Draft flow
   Show saved draft or entry flow so the app does not read as “standings only.”

## Detailed screenshot capture plan

Use a real iPhone build of the app. Do not use desktop browser captures or mocked device frames.

### Shot 1: Home

- Goal:
  Establish that bracketball is a real mobile app, not just standings pages.
- Show:
  - the app header/mobile shell
  - active navigation
  - live product context
  - enough meaningful content that the screen feels alive
- Avoid:
  - empty states
  - setup/debug banners
  - stale data or unfinished pools

### Shot 2: Pools

- Goal:
  Show that users can manage real pools and move through the app naturally.
- Show:
  - at least one joined pool card
  - the `discover & join` entry point visible on screen
  - clean, readable pool names
- Avoid:
  - too many half-finished pools
  - cluttered or repetitive card states

### Shot 3: Leaderboard

- Goal:
  Make the competitive core of the app obvious right away.
- Show:
  - the pool name
  - meaningful standings with several players
  - visible scores/ranks
  - `Share Invite` visible if it fits cleanly
- Best seeded data:
  - at least 4-6 entries
  - non-trivial score differences
  - realistic display names

### Shot 4: Bracket

- Goal:
  Show the game structure and bracket-following experience.
- Show:
  - a populated bracket
  - visible advancement paths
  - enough context to understand it at a glance
- Avoid:
  - an empty or mostly unfilled bracket
  - a crop that makes the bracket unreadable

### Shot 5: Profile

- Goal:
  Prove native/mobile-specific value and account polish.
- Show:
  - avatar
  - profile details
  - push notifications section
- Optional:
  - use the enabled-notifications state if it looks strong and clean

### Shot 6: Draft flow

- Goal:
  Show that the app includes real draft creation/entry flow, not just viewing results.
- Show:
  - a saved draft or active draft/entry screen
  - enough team/pick context to feel interactive
- Avoid:
  - confusing editing states
  - too much tiny text

## Capture notes

- Keep the same account, theme, and seeded data style across all screenshots.
- Prefer one polished review/demo account over many inconsistent accounts.
- Use realistic names and standings.
- Remove any temporary admin/debug/test UI before capture.
- If one screen looks weak, replace it with a stronger bracket or leaderboard shot rather than forcing the planned order.
